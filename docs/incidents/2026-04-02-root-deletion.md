# 事故报告：CI/CD 流水线导致根文件系统误删事故

**事故日期：** 2026-04-02
**状态：** 已解决 (服务器已重构，防护措施已强制上线)
**严重程度：** 致命 (CRITICAL) - 导致服务器基础架构全毁

## 1. 执行摘要 (Executive Summary)

在一项自动化的 GitHub Actions 部署过程中，由于极端的配置失误，导致了灾难性的服务器损毁。部署工作流错误地结合了 `root` 账户权限、根目录路径 (`/`) 以及 `rsync --delete` 指令。这导致 `rsync` 尝试将整个服务器根目录“镜像化”为仓库内容，从而将不属于仓库的所有系统关键文件（包括 `/bin`, `/etc`, `/usr`, `/lib` 等）全部永久删除。服务器瞬间瘫痪且无法修复，最终不得不通过彻底重装系统并恢复数据来解决。

## 2. 事故时间线

- **配置变更：** GitHub Secrets 中的环境变量 `DEPLOY_PATH` 被误设为 `/`。
- **触发部署：** 代码推送到 `master` 分支，触发 `deploy.yml` 工作流。
- **权限获取：** SSH 连接以 `root` 用户身份成功登录。
- **灾难执行：** `rsync -az --delete` 命令开始执行，目标指向 `/`。
- **系统瓦解：** 同步过程中，服务器操作系统赖以生存的库文件和执行文件被视为“仓库中不存在的冗余文件”而被清除。
- **服务中断：** SSH 连接因系统关键组件丢失而中断。服务器所有响应终止。
- **紧急恢复：** 确认为不可恢复损坏。销毁实例并重新构建。

## 3. 根本原因分析 (Root Cause Analysis)

本次事故是由两个核心风险点叠加形成的“完美风暴”：

1. **最高权限滥用（关键失误）：** 部署流水线使用了 `root` 用户进行 SSH 登录。这绕过了所有 Linux 标准的权限保护机制，使得所有系统文件处于无保护状态。
2. **根目录路径配置（关键失误）：** `DEPLOY_PATH` 被设为 `/`。这意味着同步的目标不是一个应用文件夹，而是整台服务器的生命线。
3. **破坏性指令选择：** 原方案使用了 `rsync --delete`。该指令在 `/` 目录下运行时，会将系统文件视为垃圾进行清理。

## 4. 强制实施的整改与防护措施 (Mandatory Measures)

为确保此类事故不再发生，以下防护措施已作为**硬性规定**集成至流水线：

### A. 强制路径安全校验
- [x] **已实施**：`deploy.yml` 现已强制加入 `Validate deploy path safety` 步骤。在任何同步操作前，代码会进行实时扫描：

- **禁止根目录**：严禁 `DEPLOY_PATH` 设置为 `/` 或 `/*`，否则流程直接熔断。
- **禁止穿越路径**：严禁使用 `..` 符号，防止通过路径穿越绕过监管。
- **禁止系统目录**：严禁指向 `/bin`, `/etc`, `/usr`, `/var`, `/lib`, `/root` 等 20 多个系统核心资产目录。

### B. 强制部署架构升级
- [x] **已实施**：全面禁用 `rsync --delete` 方案，改为采用业界公认安全的 **原子发布 (Atomic Release) 架构**：

- **隔离解压**：代码被打包成 `tar` 包后由 `scp` 发送，并在独立的、带有时间戳/哈希命名的 releases 目录下解压。
- **原子切换**：通过软链接 (Symlink) `ln -sfn` 将 `current` 指向新版本。
- **天然安全**：`tar` 解压覆盖操作具备天然的防误删属性，即便路径有误，也绝对不会产生误删系统旧文件的行为。

### C. 强制执行最低权限原则
- [ ] **必须实施**：**严禁使用 root 用户进行部署**。

- **后续行动**：新服务器必须建立专用的部署用户（如 `deployer`），并将其权限范围严格限制在应用工作目录（如 `/var/www/` 或 `/data/`）内。

### D. 脚本级安全自检
- [ ] **必须实施**：在服务器端的 `install.sh` 脚本中增加环境自检代码。

## 5. 结论

本次事故不仅是一次技术失误，更是对自动化部署流程规范性的警示。通过重构部署逻辑和加入强制性的熔断校验，我们已将系统抗风险能力提升至生产级水平。未来，所有注入流水线的变量都将被视为“不可信输入”，必须经过严格校验方可执行。
